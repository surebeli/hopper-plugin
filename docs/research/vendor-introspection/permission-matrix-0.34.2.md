## How to read the table

| Label | Meaning |
| --- | --- |
| **observed** | A command/result in the current task history demonstrated the stated behavior. It is scoped to that invocation and environment. |
| **documented (official vendor-native)** | An official vendor page cited by the web-only task describes a native CLI or endpoint feature. It does not prove that Hopper invokes, forwards, or enforces that feature. |
| **documented (stale-doc)** | Recorded in generated Hopper 0.32.0 `DISPATCH.md`; it needs 0.34.2 source/static-capability confirmation. |
| **policy** | Orchestration-lane rule recorded in the original review environment’s `AGENTS.md`; it limits use there even when an adapter is registered. |
| **unknown / unverified** | No admissible evidence establishes the behavior. It must not be assumed from a similar vendor or from a successful dispatch. |

`Strict read-only` means an enforceable vendor/CLI permission boundary, not merely a
prompt asking the model not to write. `Write + read` refers to the middle
`workspace-write` lane; it is intentionally separate from full-access flags.
URL fetch and WebSearch are also separate: evidence for one does not establish the
other.

## Cross-vendor path boundary conclusion

Every vendor has a **subject-root evidence gap**. The available material identifies
some cwd/additional-directory switches, but does not prove that the Hopper 0.34.2
dispatcher rejects a path outside the subject root, nor that a vendor's own tools
will refuse arbitrary absolute paths. A cwd flag is not a security boundary.

Therefore, until the 0.34.2 adapter and dispatcher static contracts are readable,
select a vendor only when its *vendor-native* read-only control is known to be
enforceable and keep the requested root explicit in the task. Treat all unverified
path behavior as advisory; do not extend the root implicitly.

## What is and is not covered

The matrix covers all **eight registry names confirmed by the 0.34.2 stat probe**:
`codex`, `kimi`, `opencode`, `copilot`, `agy`, `grok`, `mimo`, and `claude`.
It is not proof that this list is exhaustive, because the independent reviewer
could not enumerate the vendor directory; common extra names it stat-probed were
absent, but an unprobed name cannot be ruled out.

The table is sufficient to make the following conservative routing decisions now:

1. Never treat Kimi or Grok as a strict read-only vendor.
2. Do not treat registration as evidence of workspace-write, URL fetch, WebSearch,
   authentication, or path confinement.
3. Do not use Codex or Grok where the original review-environment lane policy applies; do not use AGY or
   Mimo while the recorded gate/quarantine remains active.
4. Claude's RO source-research and alias-specific web observations are advisory,
   time-scoped facts, not a permanent alias or network-capability contract.
5. Before filling the unknown cells, use independently verified static inspection
   of the installed 0.34.2 adapters and dispatcher with C: drive read scope, or
   a future working reflection interface. In this PATH installation,
   `hopper-dispatch --rules` and `hopper-dispatch --capabilities claude` were
   attempted but exited `0` with no stdout, so they cannot currently establish
   capability. Do not weaken read-only merely to obtain web access.
