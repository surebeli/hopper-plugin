# Design: Fuse fable governance into hopper routing; reduce fable to governance-only

Date: 2026-06-17
Status: Approved (approach B)
Repos touched: `hopper-plugin` (gains governance), `fable-5-anything` (loses Mode B)

## 1. Problem

`fable-5-anything` ships two modes:

- **Mode A (governance-only):** inline a portable behavioral constitution into a
  project's `AGENTS.md`/`CLAUDE.md` (and Kimi skill / opencode.json), so any host
  that auto-loads those charters is governed.
- **Mode B (full = governance + dispatch):** a dispatch/executor layer
  (`init`/`install`/`build-prompt`/`smoke`/`run`/`doctor`, a handoff contract,
  per-runtime adapters, `.fable/` scaffold, shims) that assembles
  `constitution + runtime-adapter + handoff` and spawns `opencode`.

`hopper-plugin` is a mature vendor-neutral **background dispatch** system over a
`.hopper/` file protocol (multi-vendor routing, background jobs, progress events,
dashboard, `host != vendor`, sandbox policy, generated dispatch-rules contract).

Mode B's dispatch mechanics are a **strict subset** of hopper (fable is
opencode-only, foreground, single-shot). The only capability Mode B has that
hopper lacks is **governance-injection**: fable prepends its constitution to the
dispatched prompt. Hopper's `composePrompt(frame, spec)` injects task-shape +
spec only, and hopper deliberately **isolates** vendors from host governance
(`codex` → `project_doc_max_bytes=0` + isolated `CODEX_HOME`; `claude` → `--bare`;
frames are anti-persona-checked).

Goal: move the *governance capability* into hopper by **fusing it into hopper's
routing** (approach B), then delete Mode B from fable so fable becomes
governance-only. Hopper stays the single dispatcher; fable stays the single
owner of the constitution text.

## 2. Key insight enabling the fusion

Fable selects its overlay by **runtime** (`opencode`/`codex`/`kimi`/`grok`/
`copilot`/`generic`). Hopper's router resolves a **vendor**
(`opencode`/`codex`/`kimi`/`grok`/`copilot`/`agy`/`mimo`/`claude`). These are the
same set of CLIs. Hopper's `resolveVendor()` already computes the exact key fable
uses. So fusion = resolve a **governance overlay off the vendor hopper already
resolved**, and inject it as a preamble inside `composePrompt()`.

**Isolation-first is complementary, not contradictory.** Hopper strips the host's
*ambient* config from the vendor; B injects the project's *chosen* governance as
explicit task payload. Different channels. Injection is in fact the only way to
govern a vendor hopper has deliberately cut off from host files. B is strictly
opt-in, so it never makes governance ambient.

## 3. Architecture (hopper side — approach B)

### 3.1 New routed artifact: `.hopper/GOVERNANCE.md`

Governance is routed the same way vendors are: a `.hopper/` markdown table with a
global default + per-row override, seeded/regenerated by a command (like
`DISPATCH.md` is generated from adapters). Shape:

- A global `constitution:` pointer. **Default = hopper's own vendored copy
  `.hopper/governance/portable-agent-core.md`**, a provenance/version-stamped
  snapshot pulled from fable's upstream `prompts/portable-agent-core.md` by
  `--init-governance --from` (generated artifact, like `DISPATCH.md`). It may
  instead point at `.fable/portable-agent-core.md` when `fable opencode setup`
  produced one. Note: host-agnostic `fable governance` inlines the core into
  `AGENTS.md` and creates **no** standalone file, so a pointer into `AGENTS.md`
  is not assumed — the vendored copy is the reliable source. Fable's
  `prompts/portable-agent-core.md` remains the single upstream source of truth.
- A `vendor → overlay` table: optional per-vendor overlay prose (the subordination
  clause: "core wins on conflict; adapter mechanics don't expand permissions;
  don't borrow another runtime's tool names"). Vendors with no row get
  constitution-only.
- Absent file ⇒ governance disabled ⇒ flow identical to today's hopper.

### 3.2 New resolver: `resolveGovernance({ hopperDir, vendor, task })`

Lives beside `resolveVendor` (new module `cli/src/governance.js`). Returns
`{ constitution: string, overlay: string } | null`.

- Reads `.hopper/GOVERNANCE.md`; if absent → `null`.
- Resolves the constitution text from the configured pointer (read once per
  dispatch; never spawns).
- Selects the overlay by `vendor` key.
- Per-queue-row override: a new optional `Govern` column in `queue.md` →
  `off` (force `null` for this task) or a named profile (future).

### 3.3 Injection point: `composePrompt(frame, spec, { governance })`

One added optional argument. Layering:

```
[constitution]            ← global behavioral contract (highest level; "core wins")
[per-vendor overlay]      ← optional subordination/mechanics-discipline prose
frame                     ← task-type SHAPE (unchanged; anti-persona-checked)
## Task spec
spec                      ← this task's detail (unchanged)
```

When `governance` is `null`, output is byte-identical to today.

### 3.4 Wiring into the dispatch flow

In `cli/src/dispatch.js::resolveDispatch`, after `resolveVendor` (vendor known)
and `loadTaskSpec`:

```
const governance = await resolveGovernance({ hopperDir, vendor, task });
const composedPrompt = composePrompt(frame, taskSpec, { governance });
```

Everything downstream is **unchanged**: `getAdapter`, `resolveAdapterOptsForTask`
(sandbox/reasoning), `envPreflight`, `adapter.args(composedPrompt, …)` / stdin,
`runSubprocessOnce` (one spawn, idle+ceiling, no retry), `parseResult`, output
capture. Governance rides inside `composedPrompt` to the vendor regardless of
arg-vs-stdin channel. Exactly one injection point.

### 3.5 New command: `hopper-dispatch --init-governance [--from <core-path>]`

Seeds `.hopper/GOVERNANCE.md` (constitution pointer + a `vendor → overlay` table
templated for the registered adapters). `--from <path>` vendors a synced copy of
fable's `prompts/portable-agent-core.md` to
`.hopper/governance/portable-agent-core.md` with a provenance/version header, and
sets that as the `constitution:` pointer (the default, standalone-safe source).
Without `--from`, `GOVERNANCE.md` is seeded with a placeholder pointer and a
comment telling the user to run `--from` or point at a `.fable/` core; a dispatch
with governance enabled but an unresolvable pointer fails fast with that guidance
(never silently dispatches ungoverned). Integrated as an optional extra in
`scaffold.js` (NOT written by default `--init-tasks`, to preserve opt-in
semantics). Mirrors the generated-not-hand-edited discipline of `DISPATCH.md`.

### 3.6 Boundaries preserved

- `verifyFrameAntiPersona` stays scoped to **frames** only; the constitution is a
  separate behavioral layer (and explicitly anti-identity), so it is not run
  through that verifier.
- No spawn added; `resolveGovernance` is pure file I/O on the `dispatch.js`
  resolve path — preserves the no-spawn-in-adapters guarantee and the
  one-dispatch = one-subprocess invariant.
- Governance is instruction (soft); hopper's `--sandbox` is enforcement (hard).
  They reinforce; governance never alters permission/timeout/routing/spawn.

## 4. Architecture (fable side — remove Mode B)

Fable becomes governance-only (Mode A). Per the overlap map:

**Kept (governance):**
- `prompts/portable-agent-core.md` (the constitution), `prompts/claude-fable-5-exclusions.md`.
- `src/charter.js` (`syncCharter`, `FABLE_BLOCK`, `FABLE_BLOCK_OPENCODE`) **plus**
  `buildInlineCharterBlock()` and `wireOpencodeGovernance()` **extracted out of
  `src/install.js` into `charter.js`** (they are governance trapped in the
  dispatch module).
- `src/skill.js` + `adapters/*.md` (governance/overlay reference prose, kept).
  `kimi.md` is embedded into the generated Kimi `SKILL.md`; the rest are
  referenced by `runtime-capabilities.json` `adapter` fields and asserted to
  exist by `test/runtime.test.js`. They document per-host overlay wiring, not
  executable dispatch, so they stay. (Optional follow-up: prune dispatch-only
  prose from `opencode.md`.)
- `src/runtime.js` + `adapters/runtime-capabilities.json`, **trimmed** of
  dispatch-only `commandSupport` (`build-prompt`/`smoke`/`run`); keep
  `injectionMode`/`hostSystemPolicy`/`charterFiles`.
- `src/mcp.js` **slimmed to `fable_runtime` only** (governance metadata);
  `fable_build_prompt`/`fable_doctor` removed (they depended on Mode B).
- `src/config.js` reduced to `PKG_ROOT` + `resolveCorePath`; `src/version.js`,
  `bin/fable.js`, gutted `src/cli.js`.
- Residual CLI: `governance`, `charter sync`, `runtime`, `codex|kimi|copilot|grok
  setup`, `opencode setup`, `mcp-server`, `help`, `version`.

**Deleted (dispatch / redundant):**
- `src/opencode.js`, `src/prompt.js`, `src/handoff.js`, `src/doctor.js`,
  `src/install.js` (after extracting its two governance fns).
- `dispatch/` (and `dispatch/prompt-assembly.md`).
- `examples/`, `scripts/demo-x-agents.ps1`.
- CLI commands: `init`, `install`, `build-prompt`, `smoke`, `run`, `doctor` (+
  `requireOpencode`/`resolveHandoffPath` helpers).

**Docs updated (explicitly including READMEs):**
- `fable-5-anything/README.md`: drop the two-mode framing and all Mode 2 sections
  (install/quickstart/commands table/footprint); reframe as governance-only;
  point dispatch users at hopper-plugin.
- `fable-5-anything/AGENTS.md` + `CLAUDE.md`: drop the "ASK FIRST: Mode 1 vs Mode
  2" block; governance-only instructions remain.
- `docs/install-modes.md`: removed or rewritten to a single-mode page.
- `package.json`: drop `dispatch/` from `files[]`; prune now-unused keywords.
- `hopper-plugin/README.md`: add a Governance section (the opt-in
  `GOVERNANCE.md` overlay + `--init-governance`).

## 5. Out of scope (deferred)

- Handoff 4-section validator as a pre-dispatch gate (was approach C).
- Task-type-keyed governance table (the `Govern` column covers the need in v1).
- Unifying fable host-setup commands with hopper host wrappers (they target the
  same CLIs but serve different roles; documented, not merged).
- Folding `runtime-capabilities` governance metadata into hopper's vendor
  registry (approach C depth).

## 6. Risks / mitigations

- **Philosophical conflict with isolation-first** → mitigated by opt-in + the
  "ambient vs chosen, host-config vs prompt-payload" carve-out; documented.
- **Constitution drift across repos** → mitigated by *reference, not fork*; the
  vendored standalone copy carries a provenance/version stamp.
- **`install.js` extraction hazard** (governance fns inside the dispatch module;
  `cli.js` eager-imports all modules) → extract `buildInlineCharterBlock` +
  `wireOpencodeGovernance` to `charter.js` and rewire imports **before** deleting
  `install.js`; `fable governance` + `fable opencode setup` must keep working.
- **Slimmed MCP erosion** → `fable_runtime` retained so `codex/grok/copilot setup`
  `mcp add fable` still advertises a real governance tool.
- **Stale dispatch vocab in `runtime-capabilities.json`** → trim
  `commandSupport` entries on the governance side.

## 7. Testing strategy

Hopper (TDD, `node --test tests/unit/*.test.js`):
- `governance.test.js`: `GOVERNANCE.md` parse (constitution pointer + vendor
  overlay table + `Govern` override); absent file → `null`; constitution read
  from pointer; vendored-copy path.
- `composePrompt` layering: with/without governance; `null` ⇒ byte-identical to
  pre-change output (regression lock).
- `resolveDispatch` integration: governance resolved between vendor + compose; no
  added subprocess (extend the existing subprocess-spawn-count guarantee test).
- `scaffold`/`--init-governance`: writes `GOVERNANCE.md`; default `--init-tasks`
  does NOT (opt-in preserved).

Fable:
- After removal: `fable governance`, `fable opencode setup`, `fable kimi setup`,
  `fable charter sync`, `fable runtime`, `mcp-server` (fable_runtime) still pass;
  removed commands error cleanly; `node --test test/*.test.js` green (delete/port
  Mode B tests).

## 8. Acceptance

1. Existing hopper behavior unchanged when no `.hopper/GOVERNANCE.md` (regression
   test proves byte-identical composed prompt).
2. With `GOVERNANCE.md`, a dispatched vendor's prompt begins with the
   constitution + its vendor overlay; `Govern: off` disables per task.
3. No new subprocess spawned by governance (spawn-count test holds).
4. Fable has no Mode B: dispatch files/commands/adapters/examples removed,
   governance commands still work, README/AGENTS/CLAUDE/install-modes reflect
   governance-only.
5. Constitution text exists only in fable (hopper references or carries a stamped
   copy).
